# 持久化目标 (`/goal`)

`/goal` 为 Hermes 设定一个跨回合持续存在的长期目标。每个回合结束后，一个轻量级的评判模型会检查助手的最后一次回复是否满足了目标。如果没有，Hermes 会自动将一条继续执行的提示词反馈到同一个会话中并继续工作——直到目标达成、你暂停或清除了目标，或者回合预算用完为止。

这是我们对 **Ralph 循环** 的实现，直接受到 Eric Traut (OpenAI) 开发的 [Codex CLI 0.128.0 的 `/goal`](https://github.com/openai/codex) 启发。其核心理念——让目标在多个回合中保持活跃，直到达成为止——是他们的。这里的实现是独立的，并适配了 Hermes 的架构。

## 何时使用

当你希望 Hermes 自行迭代而无需你每回合都重新提示时，使用 `/goal`：

- “修复 `src/` 中的所有 lint 错误，并确保 `ruff check` 通过”
- “从仓库 Y 移植功能 X，包括测试，并使 CI 通过”
- “调查会话 ID 为何有时在运行中压缩时发生漂移，并撰写报告”
- “构建一个小型 CLI 来根据 EXIF 日期重命名文件，然后针对 photos/ 文件夹进行测试”

Agent 执行一个回合就停止的任务不需要 `/goal`。那些**否则你需要说三次‘继续’**的任务，才是这个功能大显身手的地方。

## 快速开始

```
/goal 修复 tests/hermes_cli/ 中所有失败的测试，并确保 scripts/run_tests.sh 在该目录下通过
```

你将看到：

1.  **目标已接受** — `⊙ 目标已设定 (20回合预算): <你的目标>`
2.  **回合 1 运行** — Hermes 开始工作，就像你将目标作为普通消息发送一样。
3.  **评判运行** — 回合结束后，评判模型决定 `完成` 或 `继续`。
4.  **必要时触发循环** — 如果 `继续`，你将看到 `↻ 继续向目标前进 (1/20): <评判理由>`，并且 Hermes 会自动执行下一步。
5.  **终止** — 最终你会看到 `✓ 目标已达成: <理由>` 或 `⏸ 目标已暂停 — 已使用 N/20 回合`。

## 命令

| 命令 | 功能 |
|---|---|
| `/goal <文本>` | 设置（或替换）长期目标。立即启动第一个回合，因此你无需发送单独的消息。 |
| `/goal` 或 `/goal status` | 显示当前目标、其状态和已使用的回合数。 |
| `/goal pause` | 停止自动继续循环，但不清除目标。 |
| `/goal resume` | 恢复循环（将回合计数器重置为零）。 |
| `/goal clear` | 完全删除目标。 |

在 CLI 和所有消息网关平台（Telegram、Discord、Slack、Matrix、Signal、WhatsApp、SMS、iMessage、Webhook、API 服务器和 Web 仪表板）上的工作方式完全相同。

## 行为细节

### 评判模型

每个回合结束后，Hermes 会调用一个辅助模型，输入包括：

- 长期目标文本
- Agent 最近一次最终回复（最后约 4 KB 文本）
- 一个系统提示词，指示评判模型以严格的 JSON 格式回复：`{"done": <布尔值>, "reason": "<一句话理由>"}`

评判模型被设计为**保守**：只有当回复**明确**确认目标完成、最终交付物已清晰产出，或者目标无法实现/被阻塞时（视为 DONE 并附带阻塞原因，以免在不可能的任务上浪费预算），它才会将目标标记为 `done`。

### 故障开放语义

如果评判模型出错（网络波动、响应格式错误、辅助客户端不可用），Hermes 会将裁决视为 `continue` —— 一个损坏的评判模型永远不会阻碍进展。**回合预算**是真正的最后防线。

### 回合预算

默认是 20 个继续回合（`config.yaml` 中的 `goals.max_turns`）。当预算用完时，Hermes 会自动暂停，并明确告知你如何继续：

```
⏸ 目标已暂停 — 已使用 20/20 回合。使用 /goal resume 继续，或使用 /goal clear 停止。
```

`/goal resume` 会将计数器重置为零，因此你可以分阶段继续。

### 用户消息始终优先

当目标处于活动状态时，你发送的任何真实消息都比继续循环具有更高的优先级。在 CLI 上，你的消息会进入 `_pending_input`，排在排队的继续操作之前；在网关上，它同样通过适配器的 FIFO 队列处理。在你的回合结束后，评判模型会再次运行——因此，如果你的消息恰好完成了目标，评判模型会捕捉到并停止。

### 运行中的安全性（网关）

当 Agent 已经在运行时，`/goal status`、`/goal pause` 和 `/goal clear` 可以安全运行——它们只影响控制平面状态，不会中断当前回合。在运行中设置**新**目标（`/goal <新文本>`）会被拒绝，并提示你先执行 `/stop`，这样旧的继续操作就不会与新的发生竞争。

### 持久化

目标状态存储在 `SessionDB.state_meta` 中，键为 `goal:<session_id>`。这意味着 `/resume` 会从你离开的地方继续——设定一个目标，关闭笔记本电脑，明天回来，执行 `/resume`，目标仍然保持你离开时的状态（活动、暂停或完成）。

### 提示词缓存

继续提示词是一个附加到历史记录中的普通用户角色消息。它**不会**修改系统提示词、交换工具集或以任何使 Hermes 提示词缓存失效的方式触及对话。运行一个 20 回合的目标，在缓存成本上与 20 回合的正常对话相同。

## 配置

添加到 `~/.hermes/config.yaml`：

```yaml
goals:
  # Hermes 自动暂停并提示你 /goal resume 之前的最大继续回合数。
  # 默认 20。如果你想要更紧密的循环，可以降低此值；
  # 对于长时间运行的重构任务，可以提高此值。
  max_turns: 20
```

### 选择评判模型

评判模型使用 `goal_judge` 辅助任务。默认情况下，它解析为你配置的主模型（参见[辅助模型](/docs/user-guide/configuration#auxiliary-models)）。如果你想将评判模型路由到一个廉价快速的模型以降低成本，可以添加一个覆盖配置：

```yaml
auxiliary:
  goal_judge:
    provider: openrouter
    model: google/gemini-3-flash-preview
```

评判模型的调用量很小（约 200 个输出 Token），并且每回合运行一次，因此通常选择廉价快速的模型是合适的。

## 示例演练

```
你: /goal 创建四个文件 /tmp/note_{1..4}.txt，每回合一个，每个文件包含其编号作为文本

  ⊙ 目标已设定 (20回合预算): 创建四个文件 /tmp/note_{1..4}.txt，每回合一个，每个文件包含其编号作为文本

Hermes: 现在创建 /tmp/note_1.txt。
  💻 echo "1" > /tmp/note_1.txt   (0.1s)
  我已创建 /tmp/note_1.txt，内容为 "1"。按照你的要求，我将在下一回合继续创建剩余的文件。

  ↻ 继续向目标前进 (1/20): 仅创建了 4 个文件中的 1 个；剩余 3 个文件。

Hermes: [继续向你的长期目标前进]
  💻 echo "2" > /tmp/note_2.txt   (0.1s)
  已创建 /tmp/note_2.txt。还剩两个。

  ↻ 继续向目标前进 (2/20): 已创建 2/4 个文件；剩余 2 个。

Hermes: [继续向你的长期目标前进]
  💻 echo "3" > /tmp/note_3.txt   (0.1s)
  已创建 /tmp/note_3.txt。

  ↻ 继续向目标前进 (3/20): 已创建 3/4 个文件；剩余 1 个。

Hermes: [继续向你的长期目标前进]
  💻 echo "4" > /tmp/note_4.txt   (0.1s)
  所有四个文件都已创建：/tmp/note_1.txt 到 /tmp/note_4.txt，每个文件包含其编号。

  ✓ 目标已达成: 所有四个文件都已创建并包含指定内容，目标完成。

你: _
```

四个回合，一次 `/goal` 调用，你无需发出任何“继续”提示。

## 当评判模型出错时

没有评判模型是完美的。需要注意两种故障模式：

**假阴性 — 评判模型说继续，但目标实际上已完成。**
回合预算会捕获这种情况。你会看到 `⏸ 目标已暂停`，然后可以执行 `/goal clear` 或直接发送新消息。

**假阳性 — 评判模型说完成，但工作尚未完成。**
你会看到 `✓ 目标已达成`，但你知道并非如此。发送后续消息以继续，或者更精确地重新设定目标：`/goal <更具体的文本>`。评判模型的系统提示词被设计为保守，以使假阳性比假阴性更少见。

如果你觉得评判模型的裁决不可信，`↻ 继续向目标前进` 或 `✓ 目标已达成` 行中的理由文本会明确告诉你评判模型看到了什么。这通常足以诊断是目标文本模糊，还是模型的回复有问题。

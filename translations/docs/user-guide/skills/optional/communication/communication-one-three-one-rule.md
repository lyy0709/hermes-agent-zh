# One Three One 规则

用于技术提案与权衡分析的结构化决策框架。当用户在多种方法（架构决策、工具选择、重构策略、迁移路径）之间面临选择时，此技能会生成 1-3-1 格式的回复：一个清晰的问题陈述、三个各具优缺点且不同的选项，以及一个包含完成定义和实施计划的具体建议。当用户要求“1-3-1”、说“给我几个选项”或需要在竞争性方案之间做出选择时使用。

# 1-3-1 沟通规则

当任务存在多种可行方法且用户需要明确建议时使用的结构化决策格式。生成简洁的问题框架、三个包含权衡的选项以及针对推荐路径的可执行计划。

## 使用时机

- 用户明确要求“1-3-1”格式的回复。
- 用户针对技术决策说“给我几个选项”或“我有哪些选择”。
- 任务存在多种具有重要权衡的可行方法（架构、工具、迁移策略）。
- 用户需要一个可以转发给团队或利益相关者的提案。

**不要**用于只有一个明显答案的简单问题、调试会话或用户已决定采用某种方法的任务。

## 流程

1.  **问题**（一句话）
    - 用一句简洁的话陈述核心决策或期望结果。
    - 关注*是什么*，而非*如何做*——不包含实现细节、工具名称或特定技术。
    - 保持简洁。如果需要用“和”，说明你在描述两个问题。

2.  **选项**（恰好三个）
    - 提出三个不同的、可行的方案，标记为 A、B、C。
    - 每个选项都包含简要描述、优点和缺点。
    - 选项应代表真正不同的策略，而非同一方法的微小变体。

3.  **建议**（一个选项）
    - 说明你推荐哪个选项以及原因，基于用户的上下文和优先级。
    - 直接了当——这是你的专业判断，不是模棱两可的表述。

4.  **完成定义**
    - 列出推荐选项的具体成功标准。
    - 这些是具体的、可验证的结果——不是模糊的期望。
    - 如果用户选择了不同的选项，则修改此部分以匹配。

5.  **实施计划**
    - 执行推荐选项的具体步骤。
    - 在适用时包含具体的命令、工具或操作。
    - 如果用户选择了不同的选项，则修改此部分以匹配。

## 验证

- 回复恰好有一句问题陈述。
- 回复恰好有三个选项（A、B、C），每个都有优缺点。
- 回复有一个单一的建议，选择一个选项并附上推理。
- 完成定义和实施计划与推荐选项一致。
- 如果用户选择了不同的选项，则相应地更新建议、完成定义和实施计划。

## 示例

用户：“给我一个为我们的 API 客户端添加重试逻辑的 1-3-1 方案。”

Agent：

**问题：** 在负载下，对外部服务的 API 调用会间歇性失败，需要优雅的重试处理。

**选项：**

-   **选项 A：在共享工具中使用带抖动的指数退避。**
    优点：简单、广为人知的模式；所有调用者共享单一实现。
    缺点：没有熔断机制；调用者在重试期间仍然会阻塞。

-   **选项 B：使用重试库（tenacity）作为每个端点调用的装饰器。**
    优点：可按端点配置；内置支持熔断器和速率限制。
    缺点：增加依赖；装饰器模式可能使控制流变得模糊。

-   **选项 C：基于队列的异步重试，包含死信处理。**
    优点：非阻塞；失败的调用被跟踪并可重放；扩展性好。
    缺点：复杂性显著增加；需要队列后端；对于低流量服务来说过于复杂。

**建议：** 选项 A。调用量不足以证明队列基础设施的合理性，而带有指数退避和抖动的共享工具可以干净地覆盖故障模式，且无需新的依赖。

**完成定义：**
- `api_retry()` 工具处理瞬态 HTTP 错误（429、502、503、504）。
- 最多重试 3 次，使用指数退避和抖动。
- 所有外部 API 调用都通过重试工具进行路由。
- 记录每次重试尝试，包含端点和尝试次数。
- 测试覆盖重试耗尽、立即成功以及瞬态失败后成功等场景。

**实施计划：**
1.  创建 `utils/api_retry.py`，包含可配置的最大重试次数、基础延迟和可重试状态码。
2.  使用 `random.uniform(0, base_delay)` 添加抖动以防止惊群效应。
3.  用重试工具包装 `api_client.py` 中现有的 API 调用。
4.  添加单元测试，模拟每种重试场景的 HTTP 响应。
5.  使用针对不稳定端点模拟的简单压力测试在负载下进行验证。

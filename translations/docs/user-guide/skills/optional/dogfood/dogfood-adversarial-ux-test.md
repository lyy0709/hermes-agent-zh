# 对抗性用户体验测试

扮演你产品最糟糕的用户——那个讨厌技术、不想要你的软件、会找出各种理由抱怨的人。然后通过务实性过滤层筛选他们的反馈，将真实的用户体验问题与“我讨厌电脑”之类的噪音区分开。

可以把它看作一个自动化的“老妈测试”——不过是愤怒版的。

## 为什么有效

大多数 QA 发现的是 Bug。这个技能发现的是**摩擦**。一个技术正确的应用对真实用户来说仍然可能无法使用。对抗性角色能捕捉到：
- 对开发者有意义但对用户来说令人困惑的术语
- 完成基本任务所需的步骤过多
- 缺少引导或“顿悟时刻”
- 无障碍性问题（字体大小、对比度、点击目标）
- 冷启动问题（空状态、没有演示内容）
- 阻碍转化的付费墙/注册摩擦

**务实性过滤器**（第 3 阶段）是让这个技能有用而非仅仅是娱乐的关键。没有它，你可能会因为爷爷搞不懂 PDF 而在每个屏幕都添加“打印此页”按钮。

## 使用方法

告诉 Agent：
```
"在 [URL] 上运行对抗性用户体验测试"
"扮演一个脾气暴躁的 [角色类型] 来测试 [应用名称]"
"对我的预发布网站进行混蛋用户测试"
```

你可以提供一个角色，或者让 Agent 根据你产品的目标受众生成一个。

## 第 1 步：定义角色

如果未提供角色，通过回答以下问题生成一个：

1.  **谁是这个产品最棘手的用户？**（50 岁以上，非技术性职位，有数十年“老方法”经验）
2.  **他们的技术舒适度如何？**（越低越好——只用 WhatsApp、纸质笔记本、妻子帮忙设置电子邮件）
3.  **他们需要完成的那一件事是什么？**（他们的核心工作，而不是你的功能列表）
4.  **什么会让他们放弃？**（点击太多、行话、速度慢、令人困惑）
5.  **他们沮丧时会怎么说话？**（直率、骂骂咧咧、不屑一顾、叹气）

### 好的角色示例
> **“大个子米克”·麦卡利斯特** — 58 岁的力量与体能教练。只用 WhatsApp，仅此而已。他的“电子表格”是纸质笔记本。“如果我 10 秒内搞不明白，我就回去用我的笔记本。”需要记录 25 名球员的训练结果。讨厌小字、行话和密码。

### 坏的角色示例
> “一个不喜欢这个应用的用户”——太模糊，没有约束，没有声音。

角色必须**足够具体，以便在 20 分钟的测试中保持角色一致性**。

## 第 2 步：成为那个混蛋（以角色身份浏览）

1.  阅读任何可用的项目文档以了解应用上下文和 URL
2.  **完全融入角色**——他们的挫败感、局限性、目标
3.  使用浏览器工具导航到应用
4.  **尝试角色实际要完成的任务**（而不是功能游览）：
    - 他们能完成他们想做的事吗？
    - 完成它需要多少次点击/多少个屏幕？
    - 什么让他们感到困惑？
    - 什么让他们生气？
    - 他们在哪里迷路？
    - 什么会让他们放弃并回到老方法？
5.  测试这些摩擦类别：
    - **第一印象**——他们甚至愿意浏览落地页之后的内容吗？
    - **核心工作流**——他们最常需要做的那一件事
    - **错误恢复**——当他们做错事时会发生什么？
    - **可读性**——文本大小、对比度、信息密度
    - **速度**——感觉比他们当前的方法快吗？
    - **术语**——有没有他们无法理解的行话？
    - **导航**——他们能找到回去的路吗？他们知道自己在哪里吗？
6.  对每个痛点进行截图
7.  在每个页面检查浏览器控制台是否有 JS 错误

## 第 3 步：抱怨（以角色身份撰写反馈）

**以角色的身份**撰写反馈——用他们的声音，带着他们的挫败感。这不是 Bug 报告。这是一个真实的人在发泄。

```
[角色姓名] 对 [产品] 的评价

总体：[他们会继续使用吗？是/否/有条件地可能]

好的方面（勉强承认）：
- [即使是他们也不得不承认有效的东西]

坏的方面（合理的用户体验问题）：
- [会阻止他们使用产品的真实问题]

丑陋的方面（致命问题）：
- [会让他们立即卸载/取消订阅的事情]

具体抱怨：
1.  [页面/功能]：“[以角色口吻引用]”——[发生了什么，期望什么]
2.  ...

最终裁决：“[总结他们体验的一句话角色引语]”
```

## 第 4 步：务实性过滤器（关键——切勿跳过）

**跳出角色**。以产品人员的身份评估每个抱怨：

-   **红色：真正的用户体验 Bug**——任何用户都会遇到这个问题，不仅仅是脾气暴躁的。修复它。
-   **黄色：有效但优先级低**——真实问题，但只针对极端用户。记录下来。
-   **白色：角色噪音**——“我讨厌电脑”的言论，不是产品问题。忽略它。
-   **绿色：功能请求**——隐藏在抱怨中的好主意。考虑一下。

### 过滤标准
1.  一个 35 岁、有能力但忙碌的用户会有同样的抱怨吗？→ 红色
2.  这是一个真正的无障碍问题吗（字体大小、对比度、点击目标）？→ 红色
3.  这是“我希望它像纸一样工作”的对数字化的抗拒吗？→ 白色
4.  这是角色偶然发现的真实工作流低效问题吗？→ 黄色或红色
5.  修复这个问题会给 80% 感觉良好的人增加复杂性吗？→ 白色
6.  这个抱怨是否揭示了一个缺失的引导时刻？→ 绿色

**此过滤器是强制性的。** 切勿将原始的角色抱怨作为任务单发布。

## 第 5 步：创建任务单

仅针对**红色**和**绿色**项目：
-   清晰、可执行的标题
-   包含角色的原话引用（有趣且令人难忘）
-   其下的真实用户体验问题（客观的）
-   建议的修复方案（可执行的）
-   标签/标记："ux-review"

对于**黄色**项目：创建一个包含所有备注的汇总任务单。

**白色**项目仅出现在报告中。不创建任务单。

**每个会话最多 10 个任务单**——专注于最严重的问题。

## 第 6 步：报告

交付：
1.  角色抱怨（第 3 步）——有趣且直观
2.  过滤后的评估（第 4 步）——务实且可执行
3.  创建的任务单（第 5 步）——附带链接
4.  关键问题的截图

## 提示

-   **每个会话一个角色。** 不要混合视角。
-   **在第 2-3 步期间保持角色一致性。** 只在第 4 步跳出角色。
-   **首先测试核心工作流。** 不要被设置页面分心。
-   **空状态是黄金。** 新用户体验揭示了最多的摩擦。
-   **最好的发现是角色在尝试做其他事情时偶然发现的红色项目。**
-   **如果角色没有任何抱怨，说明你的角色技术太熟练了。** 让他们更年长、更没耐心、更守旧。
-   **在演示、发布之前或发布一批功能之后运行此测试。**
-   **尽可能注册为新用户。** 不要使用预先植入的管理员账户——冷启动体验是摩擦最多的地方。
-   **零个白色项目是一个信号，而不是失败。** 如果务实性过滤器没有发现噪音，说明你的产品存在真正的用户体验问题，而不仅仅是一个脾气暴躁的角色。
-   **在测试后检查项目文档中的已知问题。** 如果角色发现了一个已经在已知问题列表中的 Bug，这实际上是最具破坏性的发现——这意味着团队知道这个问题，但从未感受到用户的痛苦。
-   **订阅/付费墙测试至关重要。** 使用过期账户测试，而不仅仅是活跃账户。“当你无法支付时会发生什么”的体验揭示了产品是尊重用户还是挟持他们的数据。
-   **计算完成角色那一个任务所需的点击次数。** 如果超过 5 次，这几乎总是一个红色发现，无论角色的技术水平如何。

## 规则

-   在第 2-3 步期间保持角色一致性
-   要真正刻薄但公平——找到真正的问题，而不是捏造的问题
-   务实性过滤器（第 4 步）是**强制性的**
-   每个抱怨都需要截图
-   每个会话最多 10 个任务单
-   在预发布/已部署的应用上测试，而不是本地开发环境
-   一个角色，一个会话，一份报告
